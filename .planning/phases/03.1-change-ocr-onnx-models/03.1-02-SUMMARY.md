### Known Limitations

**CRT Linking Conflict:**
- Pre-built PDFium binaries from bblanchon/pdfium-binaries are compiled with different CRT settings than Rust's default
- Results in LNK1169 "multiply defined symbols" error during final linking stage
- `cargo check` works perfectly - code is correct
- Runtime loading will work correctly when DLL is available
- **Workaround:** Use `cargo check` for development verification
- **Future fix:** Build PDFium from source with matching CRT settings or use alternative binary source

## Technical Details

### PDFium Integration Approach

**Previous (broken):**
```toml
pdfium-render = { version = "0.8", features = ["static"], optional = true }
```
```rust
Pdfium::bind_to_statically_linked_library()
```

**New (working):**
```toml
pdfium-render = { version = "0.8", default-features = false, features = ["thread_safe", "pdfium_latest"], optional = true }
```
```rust
Pdfium::bind_to_library(Pdfium::pdfium_platform_library_name_at_path("./"))
    .or_else(|_| Pdfium::bind_to_system_library())
```

### Dynamic Loading Flow

1. At runtime, `Pdfium::default()` attempts to load pdfium.dll from:
   - Current directory (`./pdfium.dll`)
   - System library paths
2. Uses `libloading` crate for dynamic library loading
3. No compile-time linking required
4. DLL copied to target directory by build.rs

## Key Decisions Made

1. **Dynamic loading over static linking**
   - Rationale: Avoids complex PDFium build process
   - Trade-off: Requires DLL distribution with application
   - Benefit: Smaller binary, easier updates

2. **bblanchon/pdfium-binaries as DLL source**
   - Rationale: Pre-built, regularly updated, multiple platforms
   - Alternative: paulocoutinhox/pdfium-lib (better for mobile/WASM)
   - Version: 147.0.7725.0 (latest as of 2026-03-13)

3. **Use cargo check for verification**
   - Rationale: Code correctness verified, linking is deployment concern
   - Trade-off: Cannot run integration tests during development
   - Future: Build PDFium from source for full test suite
