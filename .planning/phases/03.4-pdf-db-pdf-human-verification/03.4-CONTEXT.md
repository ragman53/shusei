# Phase 03.4: PDF Database Integration for Human Verification - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning
**Source:** Code analysis and debugging session

<domain>
## Phase Boundary

This phase fixes a critical bug in the PDF import flow that prevents human verification of PDF loading:

**Problem:** PDF files are imported but the path is not stored in the database, making it impossible for the reader to find and convert the PDF.

**Scope:**
1. Add `pdf_path` column to books table
2. Store PDF path when creating book records
3. Update reader to use stored path for conversion
4. Verify PDF loading works end-to-end on Android

**Out of Scope:**
- OCR conversion logic (already implemented)
- New UI components
- Performance optimizations

</domain>

<decisions>
## Implementation Decisions

### Database Schema
- **LOCKED:** Add `pdf_path TEXT` column to `books` table (nullable, only set for PDF books)

### Book Creation Flow
- **LOCKED:** When creating a book from PDF import, store the `copied_path` returned from `import_pdf()`
- **LOCKED:** Use book's own ID as the UUID for the PDF filename (simpler approach)

### Reader Flow
- **LOCKED:** Reader reads `pdf_path` from database instead of guessing path
- **LOCKED:** If `pdf_path` is null or file doesn't exist, show appropriate error

### OpenCode's Discretion
- Migration strategy for existing books (if any)
- Error messages for missing PDF files

</decisions>

<specifics>
## Root Cause Analysis

### The Bug

**Current Flow (BROKEN):**
1. User imports PDF → `import_pdf()` saves to `pdfs/{uuid-a}.pdf` 
2. User confirms metadata → `create_book()` uses `{uuid-b}` as book ID
3. Reader tries to find PDF at `pdfs/{uuid-b}.pdf` → FILE NOT FOUND!

**Code References:**

`src/core/pdf.rs:199` - PDF saved with random UUID:
```rust
let uuid = uuid::Uuid::new_v4().to_string();
let copied_path = pdfs_dir.join(format!("{}.pdf", uuid));
```

`src/ui/library.rs:300` - Book created with different UUID:
```rust
let new_book = NewBook {
    id: Some(uuid::Uuid::new_v4().to_string()),  // Different UUID!
    ...
};
```

`src/ui/reader.rs:195` - Reader guesses wrong path:
```rust
let pdf_path = app_data_dir.join("pdfs").join(format!("{}.pdf", book_id));
```

### The Fix

**Option A (Simple):** Use book's ID for PDF filename
1. Generate book ID first
2. Use that ID for PDF filename
3. No need for `pdf_path` column

**Option B (Flexible):** Add `pdf_path` column
1. Keep current PDF naming (random UUID)
2. Add `pdf_path` column to books table
3. Store the path when creating book

**Recommended:** Option B - More flexible, handles edge cases (PDFs moved, renamed, etc.)

</specifics>
