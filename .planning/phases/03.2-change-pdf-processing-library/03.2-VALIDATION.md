# Phase 03.2: Change PDF Processing Library - Validation Strategy

**Phase:** 03.2
**Phase Slug:** change-pdf-processing-library
**Created:** 2026-03-13

---

## Validation Architecture

This phase replaces pdfium-render with hayro for PDF rendering. Validation focuses on:

1. **Functional correctness** — PDFs render without errors
2. **OCR accuracy** — Rendered pages produce equivalent OCR results
3. **Performance** — Batch processing completes without memory issues
4. **Error handling** — Invalid PDFs handled gracefully

---

## Validation Dimensions

### Dimension 1: Requirement Coverage

**Requirements:** PDF-02, PERF-02

| Requirement | Validation Method |
|-------------|-------------------|
| PDF-02: PDF page processing | Render all pages from test PDF, verify OCR output |
| PERF-02: UI not freezing | Batch processing (10 pages/batch), parallel rendering |

### Dimension 2: Must-Have Verification

**Truths:**
- PDFs load without CRT errors
- Pages render to RGBA bitmaps
- OCR pipeline consumes rendered pages
- Batch processing handles 100+ pages
- Errors don't crash the app

**Artifacts:**
- `src/core/pdf.rs` — New PdfProcessor with hayro
- `Cargo.toml` — hayro dependency, pdfium-render removed
- Tests for single-page and batch rendering

**Key Links:**
- `PdfProcessor::render_page()` → `hayro::Page::render()`
- `PdfProcessor::render_pages_batch()` → parallel rendering
- OCR pipeline → rendered page bitmaps
