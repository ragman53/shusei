### Task 2: Add parallel rendering with concurrency limit ✅

**Implementation:**
- Added `use rayon::prelude::*;` import
- Created dedicated ThreadPool with 3 threads: `rayon::ThreadPoolBuilder::new().num_threads(3).build()`
- Used `into_par_iter()` for parallel iteration over page range
- Concurrency limited to 3 operations to prevent memory spikes

**Pattern:**
```rust
let pool = rayon::ThreadPoolBuilder::new()
    .num_threads(3)
    .build()
    .unwrap();

let pages: Result<Vec<_>> = pool.install(|| {
    (start_idx..end_idx)
        .into_par_iter()
        .map(|i| self.render_page(document, i as u32, width, height))
        .collect()
});
```

- `PdfConversionService::convert_pdf()`: Updated to use new signature, handles DB/storage integration

## Architecture Notes

**Design Decision:** Separated rendering from storage/progress tracking

The original `render_pages_batch()` included database progress tracking and storage integration. This was refactored to:
- `render_pages_batch()`: Pure rendering with parallel processing
- `PdfConversionService::convert_pdf()`: Handles batch orchestration, DB updates, and storage

**Rationale:**
- Cleaner separation of concerns
- Easier to test rendering logic independently
- More flexible for future use cases (e.g., rendering without storage)
- Better aligns with single responsibility principle

## Performance Characteristics

**Memory Management:**
- 10-page batches prevent loading entire PDF into memory
- 3-thread concurrency limit prevents CPU/memory spikes
- Suitable for 100+ page PDFs on low-RAM devices

**Expected Performance:**
- ~3x speedup on quad-core systems vs sequential rendering
- Bounded memory usage regardless of PDF size
- Progress tracking every 10 pages (per batch)
