## Executive Summary

**hayro** is a pure Rust PDF interpreter and renderer that can replace pdfium-render, eliminating the CRT linking conflict (LNK1169) caused by PDFium's pre-built binaries. Key advantages:

- **Pure Rust** — No FFI, no external C++ dependencies, no CRT conflicts
- **No unsafe code** — Crate-level `#![forbid(unsafe_code)]`
- **CPU-only rendering** — No GPU dependencies, works on all Android devices
- **Feature-complete** — 1000+ PDF test suite from pdf.js and PDFBOX
- **Active development** — 603 GitHub stars, 1,533 commits, last update Mar 10, 2026

**Tradeoffs:**
- Performance not yet optimized (development priority)
- No encrypted/password-protected PDF support
- No blending/isolation/knockout groups support yet

**Verdict:** Excellent fit for this use case — OCR preprocessing doesn't need advanced features, pure Rust eliminates CRT issues.

### Rendering API (from docs.rs/hayro)

```rust
use hayro::{Pdf, RenderSettings};
use vello_cpu::RenderContext;

// Load PDF
let pdf_data = std::fs::read("document.pdf")?;
let pdf = Pdf::new(&pdf_data);

// Render settings
let settings = RenderSettings {
    x_scale: 1.0,      // 1x scale for OCR
    y_scale: 1.0,
    bg_color: Some([255, 255, 255, 255]), // White background
};

// Render page to pixmap
let page = pdf.get_page(0).unwrap();
let pixmap = page.render(&settings);

// Access pixel data
let pixels: &[u8] = pixmap.data();  // RGBA format
let width: u32 = pixmap.width();
let height: u32 = pixmap.height();
```

### Key Types

| Type | Purpose |
|------|---------|
| `Pdf` | Represents a loaded PDF document |
| `Page` | Single PDF page |
| `RenderSettings` | Rendering configuration (scale, background) |
| `Pixmap` | Rendered bitmap output (RGBA) |

### Error Handling

hayro uses standard Rust `Result<T, E>` pattern. Errors include:
- Invalid PDF structure
- Missing pages
- Render failures

No special error enum needed — use `hayro`'s error types directly.

## Migration from pdfium-render

### Current pdfium-render API (src/core/pdf.rs)

```rust
// Current implementation
pub struct PdfProcessor {
    pdfium: Pdfium,
}

pub fn new() -> Result<Self> {
    // Load PDFium library (CRT conflict here!)
    let bindings = Pdfium::bind_to_library(...)
    let pdfium = Pdfium::new(bindings);
    Ok(Self { pdfium })
}

pub fn open<'a>(&'a self, path: impl AsRef<Path>) -> Result<PdfDocument<'a>> {
    self.pdfium.load_pdf_from_file(path.as_ref(), None)
}

pub fn render_page(&self, document: &PdfDocument, page_index: u32, width: u32, height: u32) -> Result<Vec<u8>> {
    let page = document.pages().get(page_index as u16)?;
    let bitmap = page.render_with_config(&PdfRenderConfig::new()
        .set_target_width(width as i32)
        .set_target_height(height as i32)
    )?;
    Ok(bitmap.as_bytes().to_vec())
}
```

### Proposed hayro API

```rust
// New implementation
pub struct PdfProcessor {
    // No external library handle needed!
}

pub fn new() -> Result<Self> {
    log::info!("PDF processor initialized (hayro, pure Rust)");
    Ok(Self {})
}

pub fn open(&self, path: impl AsRef<Path>) -> Result<PdfDocument> {
    let pdf_data = std::fs::read(path.as_ref())?;
    let pdf = Pdf::new(&pdf_data);
    Ok(PdfDocument { pdf, data: pdf_data })
}

pub fn render_page(&self, document: &PdfDocument, page_index: u32, width: u32, height: u32) -> Result<Vec<u8>> {
    let page = document.pdf.get_page(page_index as usize)?;
    let settings = RenderSettings {
        x_scale: width as f32 / document.width(),
        y_scale: height as f32 / document.height(),
        bg_color: Some([255, 255, 255, 255]),
    };
    let pixmap = page.render(&settings);
    Ok(pixmap.data().to_vec())
}
```

### Key Differences

| Aspect | pdfium-render | hayro | Migration Impact |
|--------|---------------|-------|------------------|
| External dependency | PDFium C++ library | Pure Rust | ✓ Eliminates CRT conflict |
| Library loading | `Pdfium::bind_to_library()` | None | ✓ Simpler initialization |
| Page indexing | `u16` | `usize` | Minor code change |
| Render config | Builder pattern | Struct fields | Minor API change |
| Output format | RGBA bitmap | RGBA bitmap | ✓ Compatible |
| Grayscale | Not directly supported | Post-process RGBA | Add conversion step |

## Android Compatibility

### hayro on Android

**Status:** Compatible — pure Rust with no platform-specific dependencies.

**Evidence:**
- Used by Typst for PDF image export (including WASM/Android targets)
- No GPU dependencies (CPU-only rendering via `vello_cpu`)
- `#![forbid(unsafe_code)]` — safe for NDK
- No FFI calls — no JNI complexity

No special Android configuration needed. Add to `Cargo.toml`:

```toml

```

Remove pdfium-render:

## Grayscale Output

hayro renders to RGBA format. For OCR pipeline compatibility:

### Option 1: Post-process to grayscale (RECOMMENDED)

```rust
fn rgba_to_grayscale(rgba: &[u8]) -> Vec<u8> {
    rgba.chunks(4)
        .map(|pixel| {
            // Standard luminance formula
            ((pixel[0] as u32 * 299 + pixel[1] as u32 * 587 + pixel[2] as u32 * 114) / 1000) as u8
        })
        .collect()
}
```

**Pros:** Simple, matches current OCR preprocessing pipeline
**Cons:** Extra conversion step (minimal cost)

### Option 2: Keep RGBA for OCR

NDLOCR-Lite can handle RGBA input. No conversion needed.

**Recommendation:** Start with Option 2 (simpler), benchmark OCR accuracy. Convert to grayscale only if needed.
