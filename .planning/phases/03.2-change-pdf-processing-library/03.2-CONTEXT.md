# Phase 03.2: Change PDF Processing Library - Context

**Gathered:** 2026-03-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace pdfium-render with hayro for PDF page rendering to images. This phase focuses on rendering only — OCR text extraction remains handled by NDLOCR-Lite (Phase 03.1).

**In scope:**
- PDF rendering to images for OCR pipeline
- Batch processing (10 pages/batch, parallel rendering)
- Error handling and retry logic

**Out of scope:**
- OCR engine changes (Phase 03.1)
- Metadata extraction enhancements
- Annotation or form support

</domain>

<decisions>
## Implementation Decisions

### Migration Strategy
- **Clean break** — Remove pdfium-render completely, switch to hayro in one change
- **Rollback** — Rely on git revert if needed (no backup branch or dual installation)
- **Workarounds** — Implement workarounds for missing hayro features BEFORE migration
- **Validation** — Use existing 373-page test PDF from Phase 03-07

### API Design
- **Direct integration** — Call hayro::render() directly, no abstraction layer
- **Render settings** — 1x scale, grayscale output (matches current OCR preprocessing)
- **Batch processing** — Follow hayro demo design, parallel rendering preferred (faster)
- **Error handling** — Return `Result<T, hayro::Error>` from PDF operations

### Error Handling
- **Error types** — Handle invalid PDF, missing pages, and render failures
- **Retry policy** — Retry once on failure, then skip the page and continue
- **Error type** — Use hayro::Error directly, no wrapper enum

### Testing
- **First test** — Batch processing with 10+ pages
- **Validation method** — OCR accuracy on rendered pages (not visual comparison)
- **Test automation** — Add automated unit and integration tests

### OpenCode's Discretion
- Specific retry delay timing
- Log message formatting
- Test file organization

</decisions>
